#### Python Database Framework - P 65

--------------

**사용 편의성**

```
직관적인 데이터베이스 엔진과 데이터베이스 추상화 레이어를 비교할 때 후자가 더 우월하다.
추상화 레이어나 오브젝트 관계 매퍼(Object-relational mapper, ORM)
혹은 오브젝트-도큐먼트 매퍼(Object-Document Mapper, ODM)는 상위 레벨 오브젝트-기반 오퍼레이션을
하위 레벨 데이터베이스 명령어로 직관적으로 변경하도록 하는 기능을 제공한다.
```


**성능**

```
ORM과 ODM이 오브젝트 도메인을 데이터베이스 도메인으로 변환하기 위해 해야 할 변경은
오버헤드를 가질 수 밖에 없다. 이때 성능 저하는 대부분 미비한 수준이지만 항상 그런 것은 아니다.
일반적으로 ORM과 ODM을 사용하여 얻게 되는 생산성은 약간의 성능 저하를 상쇄할 만큼 뛰어나다.
따라서 성능 저하만으로 ORM과 ODM을 포기하는 것은 합리적이지 않다.
네이티브 데이터베이스 명령어로 직접 구현하여 최적화할 필요가 있는 특정한 오퍼레이션의 경우에
내부 데이터베이스에 대한 옵션 액세스를 제공하는 데이터베이스 추상화 레이어를 선택하는 것은 타당하다.
```

**호환성**

```
데이터베이스를 선택할 때는 여러분의 개발과 제품 플랫폼에 사용 가능한지를 고려해야한다.
예를 들어, 클라우드 플랫폼에 애플리케이션을 호스팅할 계획이 있다면, 이러한 서비스를 제공하는 데이터베이스가
무엇인지 찾아야 한다.

다른 호환성 측면은 ORM과 ODM에 적용되는 점이다. 이 프레임워크가 하나의 데이터베이스 엔진을 위한
추상화 레이어를 제공한다 하더라도, 다른 추상화 레이어는 데이터베이스 엔진 선택을 제공한다.
이것은 같은 오브젝트 기반 인터페이스와 함께 액세스 가능하다. 이를 잘 보여 주는 대표적인 예는
SQLAlchemy ORM이며 이것은 일반적인 MySQL, Postgre, SQLite를 포함하여 관계형 데이터베이스 엔진의
리스트를 제공한다.
```

**플라스크 통합**

```
플라스크를 사용하여 통합할 프레임워크를 선택하는 것은 필수적인 것은 아니지만,
여러분이 직접 통합 코드를 작성해야 하는 부담을 줄일 수 있다. 플라스크 통합으로 설정과 오퍼레이션을 간단하게
할 수 있으므로 플라스크 확장으로 특별히 설계된 패키지를 사용할 것을 추천한다.
```
